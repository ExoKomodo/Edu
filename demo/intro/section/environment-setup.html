Laying a foundation

TODO: Explain how the foundation is the true basics like: setting up a git repo, adding a basic README, setting up basic folders, and creating an install script for basic tools

You are never benefitted from waiting to lay your foundation. Your foundation consists of the most important thing to every codebase, the code itself. To get a tangible feeling on the code, what a developer does is they make a source control repository. Now there are a few different options out there for source control:

  • Git
  • Subversion
  • Mercurial
In my professional career, I have used subversion and git. I have never witnessed a mercurial repo, but know that they are famously used at companies like Facebook and Google.
There is not one end-all be-all solution for version control, but use whatever is most available to the context you are working in. In the end, they are going to be doing the same thing, and it is more important to initially maintain a general understanding of all things around you.
I speak more about my philosophy of general understanding over here on the Edu blog.
For each version control solution, you have a choice of different ways to interact with it. I will give you my personal experience with these choices for source control, and my honest opinion on their merits.
  • Git
    • The official git client from the git project, source code available here.
    • VSCode is a very competent git client and will help ensure installation.
    • Sublime Merge is a git client from the creators of the Sublime Text code editor
    • Github Desktop will work best if you will be using Github. This tool is not intended to be used with non-Github repos.
    • Git's suggested GUI clients
  • Subversion (mostly usable on Windows)
    • TortoiseSVN is the only client I ever used or was made aware of before making this course
    • Comparison of clients
  • Mercurial
    • I have no idea and if you find yourself using Mercurial, something else is likely wrong already.
I think the point may have already come across, but I don't generally think of using Subversion or Mercurial for source control management. I have heard of their being merits to these systems in specific cases. If you consider these are entirely different tools, the assumption tracks and you won't be surprised to learn that this is because they're designed entirely differently.

tl;dr
  • Git allows the developer to work locally without network connection. You hold a copy of the repo state on your local computer, and occasionally synchronize with the centralize git server when "pushing" changes
